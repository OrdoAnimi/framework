## Context

An Australian NGO healthcare organisation operates a distributed on-premises estate across **200 sites** serving approximately **20,000 users**. The current estate is characterised by:

- **Aging VMware infrastructure** (vSphere/vCenter generations approaching end of general support) distributed unevenly across an undetermined number of data centres and edge sites
- **No documented Capability Model** — the organisation lacks a formal baseline of what workload types, service tiers, and infrastructure capabilities exist, where they run, and what their criticality classifications are
- **Multiple data centres with no defined consolidation target** — informal consolidation intent exists but no binding technical model governs DC scope, tier assignment, or workload placement
- **Healthcare regulatory context** — the organisation is subject to the Australian Privacy Act 1988, the My Health Records Act 2012, Australian Digital Health Agency (ADHA) security frameworks, and applicable state health data obligations. Any infrastructure consolidation must preserve compliance posture throughout transition

The organisation has a stated intent to consolidate to **2–3 geo-located data centres** operating as a rationalised primary/secondary/tertiary (or primary/secondary) topology.

**The root problem this ADR addresses:**  
Before platform selection, migration sequencing, or vendor engagement, the organisation must establish a binding **Capability Model** — a structured classification of what infrastructure capabilities are required, at what tier, at what site granularity, and under what constraints. Without this, consolidation decisions will be made against an unknown baseline, producing an **Integrity Gap** between assumed and actual architecture from day one.

This is the epistemological precondition for all subsequent infrastructure decisions. An organisation cannot consolidate what it has not classified.

---

## Decision

**We will adopt a formal Capability Model as the binding technical baseline for the DC consolidation programme, structured across five capability domains, before any platform selection, DC decommission scheduling, or migration wave planning is initiated.**

The five capability domains are:

| Domain | Description |
|--------|-------------|
| **CD-1: Compute** | Virtualised and bare-metal compute capacity, VM density targets, CPU/memory envelope per tier |
| **CD-2: Storage** | Block, file, and object storage classification; tiering policy; healthcare data retention obligations |
| **CD-3: Network** | Site interconnect topology, WAN/SD-WAN classification, inter-DC replication bandwidth, clinical application latency tolerances |
| **CD-4: Identity and Access** | Directory services, privileged access management, federated identity, clinical system authentication patterns |
| **CD-5: Data Protection and Compliance** | Backup and recovery RTO/RPO per workload class, data sovereignty classification, ADHA framework alignment, encryption at rest and in transit |

Each capability domain will be classified at **three infrastructure tiers**:

| Tier | Definition |
|------|------------|
| **T1 — Core DC** | Primary geo-located DC. Hosts tier-1 clinical workloads, domain controllers, core storage, and compliance-controlled data |
| **T2 — Secondary DC** | Disaster recovery and load-bearing secondary. Must sustain T1 workload failover within declared RTO |
| **T3 — Edge/Site** | Residual site infrastructure for clinical applications requiring local compute (e.g., imaging, disconnected operation). Minimal footprint only |

The Capability Model will be codified as a **living artefact** under version control, with each capability classification carrying an owner, a site-to-tier mapping, a workload inventory reference, and a compliance flag. It will serve as the **source of truth** against which all subsequent ADRs, migration plans, and platform decisions are validated.

---

## Rationale

### Why a Capability Model before platform selection

The most common failure mode in estate consolidation programmes is platform selection driven by vendor capability rather than organisational need. Without a Capability Model, the organisation cannot evaluate whether a proposed platform satisfies actual requirements — it can only evaluate whether a vendor's feature set is impressive.

In the healthcare context, this failure is compounded by compliance risk: a platform selected without understanding the data sovereignty, clinical application latency, and access control requirements of the estate may satisfy compute economics while violating regulatory obligations. Discovering this post-commitment is a programme-ending Integrity Gap.

### Why five domains

The five capability domains (Compute, Storage, Network, Identity, Data Protection) represent the minimum set required to describe infrastructure behaviour in a healthcare estate without conflation. Each domain has distinct ownership, distinct compliance obligations, and distinct failure modes.

Collapsing these into a single "infrastructure" category is the most common source of scope ambiguity in consolidation programmes. Separating them forces explicit decisions at each domain boundary.

### Why three tiers

The T1/T2/T3 tiering model reflects the stated consolidation intent (2–3 geo-located DCs) while preserving a classification for residual edge compute. Forcing a binary T1/T2 model at this stage would prematurely foreclose the edge question — 200 sites implies some sites will retain local compute capability for clinical or connectivity reasons. A three-tier model allows that decision to be made per-site, with evidence, rather than by default.

### Why version control

The Capability Model is not a document. It is a technical baseline that will be interrogated by every downstream decision in this programme. Storing it outside version control introduces the same decay risk as any architecture documentation that lives in a shared drive: it will diverge from reality within weeks of first publication. Versioned, diff-able, owner-attributed records are the minimum standard for a baseline that is expected to evolve across a multi-year consolidation.

### Healthcare-specific constraints driving this decision

| Constraint | Implication for Capability Model |
|------------|----------------------------------|
| Australian Privacy Act 1988 | PII/health record data must be classified and tracked to specific storage tier and location |
| My Health Records Act 2012 | Workloads interacting with My Health Record infrastructure must be classified T1 minimum; data sovereignty to Australian soil mandatory |
| ADHA ISM / Essential Eight alignment | Security capability baselines must be mapped per tier; Essential Eight maturity level targets must be declared per domain |
| Clinical application availability | RTO/RPO obligations for clinical workloads require explicit T1/T2 data protection capability specification before DC decommission can be sequenced |
| Disconnected operation (site resilience) | 200 geographically distributed sites include remote/rural sites where WAN failure cannot disable clinical function — T3 tier classification must address this |

### Alternatives considered

| Alternative | Why rejected |
|-------------|--------------|
| Proceed directly to platform (hypervisor) selection | Cannot evaluate platform suitability without knowing workload classification, compliance tier, and capacity envelope. Produces procurement before architecture. |
| Lift-and-shift existing VMware to cloud (IaaS) | Does not address end-of-support risk for the VMware estate; replicates existing complexity in a more expensive form; does not reduce DC footprint or consolidate capability. Deferred, not resolved. |
| Single-DC consolidation | Violates healthcare data resilience obligations and clinical availability requirements. A single DC architecture cannot meet declared RTO for tier-1 clinical workloads at this scale. |
| Informal workload inventory (spreadsheet, ad-hoc) | Produces a snapshot that cannot be validated, versioned, or tested. The Integrity Gap opens immediately. Not a Capability Model — it is a list. |
| Vendor-led assessment as Capability Model substitute | Vendor assessments reflect vendor capability, not organisational need. Creates structural bias in subsequent platform selection. Acceptable as input; not acceptable as the baseline itself. |

## Consequences

### Immediate obligations

- A **Capability Model Working Group** must be formed with named owners per capability domain (CD-1 through CD-5) before this ADR moves to Accepted status
- A **workload discovery exercise** is required to populate the model. Tooling (e.g., VMware Live Optics, RVTools, or equivalent) must be selected and deployed within **21 days of ADR acceptance** to baseline the existing VMware estate
- The Capability Model artefact must be instantiated in version control (git) with an agreed schema before discovery data is ingested — data without schema produces an inventory, not a model
- All subsequent ADRs in this programme (**ADR-002 onwards**) must reference this Capability Model as their governing baseline. CR-T2 applies.

### Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Discovery exercise is incomplete — sites not captured | High | Mandatory sign-off from each of 200 site leads confirming inventory completeness before baseline is declared |
| Capability Model treated as document, not baseline | High | Enforced by fitness functions (see below); version control is mandatory, not optional |
| VMware licensing changes accelerate before model is complete | Medium | Discovery prioritised to T1-candidate workloads first; 30-day sprint to T1 baseline minimum viable |
| Healthcare compliance obligations not fully enumerated at model creation | High | Legal/compliance review of Capability Model schema is a gate condition before discovery commences |
| Key personnel dependency — model knowledge concentrated in individuals | Medium | CR-O1 (single named owner per domain) combined with documented handover obligation |

### Integrity Gap exposure

The current state — no Capability Model, no formal DC tier assignment, no workload classification — represents the maximum Integrity Gap for this programme. Every day this ADR remains unaccepted is a day the programme operates against an unknown baseline. The 14-Day Decision Aging Rule (CR-E1) applies from the date of this record.

## Fitness Functions

All fitness functions below must be implemented in the delivery pipeline and traceable per CR-T4.

```python
# [ADR-001] — Capability Model completeness: all 200 sites must have 
# a workload inventory entry before DC consolidation wave-1 can be approved
assert capability_model.sites_inventoried() == 200

# [ADR-001] — Domain coverage: all five capability domains must have 
# a named owner and at least one classified workload entry
assert all(
    capability_model.domain(d).owner is not None and
    capability_model.domain(d).workload_count() > 0
    for d in ["CD-1", "CD-2", "CD-3", "CD-4", "CD-5"]
)

# [ADR-001] — Tier assignment: every workload in the model must carry 
# a tier classification (T1, T2, or T3); unclassified workloads block migration
assert capability_model.unclassified_workload_count() == 0

# [ADR-001] — Compliance flag coverage: every workload handling 
# health records or PII must carry an explicit compliance flag
assert capability_model.workloads_with_health_data_without_compliance_flag() == 0

# [ADR-001] — Version control: Capability Model must be committed to 
# the designated git repository; artefact not in VCS is non-compliant
assert vcs.artefact_exists("capability-model/baseline.yaml") == True

# [ADR-001] — Staleness guard: Capability Model must not exceed 
# 90 days since last validated update during active programme delivery
assert capability_model.days_since_last_validated_update() <= 90

# [ADR-001] — ADR traceability: all subsequent ADRs (ADR-002+) must 
# carry a reference to ADR-001 as governing baseline; orphaned ADRs block acceptance
assert all(
    adr.references_governing_baseline("ADR-001")
    for adr in adr_register.post("ADR-001")
    if adr.status in ["Proposed", "Accepted"]
)
```
