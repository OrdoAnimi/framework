## Context

A Government Agency operates a mature on-premises data platform centred on **IBM DataStage** (ETL/ELT pipeline orchestration) and **IBM Cognos Analytics** (enterprise reporting and BI). Both products are deployed on hardware that is approaching or has exceeded vendor support lifecycle and agency refresh thresholds. The current state exhibits the following material conditions:

**Hardware and Lifecycle Risk**
- Server hardware hosting DataStage and the underlying data warehouse is aged, with maintenance costs escalating and failure risk increasing. Replacement parts availability is constrained.
- Operating system and middleware versions are behind current vendor support windows, creating unpatched vulnerability exposure and limiting upgrade paths.
- Capacity headroom for additional workloads is effectively exhausted. New reporting initiatives are being deferred due to infrastructure constraint, not business priority.

**Operational Integrity Gap**
- The documented architecture of the on-premises environment has diverged materially from the deployed state. Configuration changes, emergency patches, and undocumented pipeline modifications have accumulated over multiple operational years. This constitutes a measurable **Integrity Gap** between the architecture description and the live system.
- No continuous enforcement of pipeline SLAs or data quality contracts exists. Fitness functions are absent. Failures surface reactively via user complaints rather than proactively via instrumented assertions.

**Vendor Product Direction**
- IBM has published a cloud-forward roadmap for both DataStage (as **IBM DataStage on IBM Cloud Pak for Data / Cloud**) and Cognos Analytics (as **IBM Cognos Analytics on Cloud**). On-premises feature parity with cloud releases is diverging, with new capabilities — including AI-assisted ETL, enhanced lineage, and automated data quality — available only on cloud-native or SaaS deployments.
- Continued investment in on-premises licensing represents an architectural trajectory that is diverging from IBM's primary development path.

**Government Agency Obligations**
- The Agency operates under obligations for data sovereignty, protective security (aligned to the Australian Government Information Security Manual or equivalent jurisdiction-specific framework), and procurement policy that governs consumption of cloud services. These obligations are not blockers to cloud adoption but constrain the permissible deployment topology (e.g., data residency must be within jurisdiction).
- An uplift event is a forcing function to resolve accumulated technical debt, re-establish architecture truth, and establish fitness function coverage that the current environment cannot support.

**Decision Trigger**
This ADR is raised because the Agency must select a target deployment model for the DataStage and Cognos platform before hardware failure forces an unplanned, uncontrolled migration. The decision has been open informally for an extended period. Per **CR-E1 (14-Day Decision Aging Rule)**, this decision requires binding resolution. Continued deferral increases both operational risk and the cost of the eventual migration.

---

## Decision

**The Agency will migrate DataStage and Cognos Analytics from on-premises infrastructure to a cloud-hosted deployment model, using IBM Cloud Pak for Data on a sovereign-compliant cloud substrate (Azure Government / AWS GovCloud / IBM Cloud for Government — jurisdiction-determined), with a phased delivery approach beginning with infrastructure decommission risk reduction and concluding with full cloud-native operational state.**

**Sub-decisions binding under this record:**

1. **DataStage** will be deployed as IBM DataStage on IBM Cloud Pak for Data (containerised, OpenShift-based), not re-platformed to a non-IBM ETL product at this stage. Re-platforming to an alternative ETL engine (e.g., Azure Data Factory, AWS Glue) is deferred and will be governed by a separate ADR if initiated.

2. **Cognos Analytics** will be migrated to IBM Cognos Analytics on Cloud (SaaS) or Cloud Pak for Data-hosted deployment — final selection to be confirmed by the Trade-off Matrix governing this ADR (CR-T2). The constraint is that report fidelity, row-level security, and existing semantic layer (Framework Manager / Data Modules) must be preserved without forced reauthoring at migration go-live.

3. **The underlying data warehouse** (source of truth for both platforms) will be assessed separately and governed by a subsequent ADR (ADR-002, to be raised). This ADR does not bind the data warehouse platform decision; it binds the DataStage and Cognos deployment model only.

4. **Lift-and-shift is the migration pattern for Phase 1.** Architectural optimisation (pipeline refactoring, semantic layer modernisation, adoption of cloud-native data quality features) is deferred to Phase 2 to reduce go-live risk and meet hardware decommission timelines.

5. **All deployed fitness functions must be in place before Phase 1 go-live.** Migration to cloud without instrumented enforcement replicates the Integrity Gap into the new environment and is not acceptable.

## Rationale

### Why cloud over on-premises refresh

An on-premises hardware refresh was evaluated. It was rejected on the following grounds:

- A like-for-like hardware refresh extends the operational life of a platform architecture that is already accumulating an Integrity Gap. It does not resolve the absence of fitness function coverage, the divergence between documented and actual state, or the approaching end of IBM's on-premises feature investment.
- Capital expenditure for a hardware refresh is a sunk cost within 5–7 years given IBM's product direction. Cloud consumption shifts this to operational expenditure with elasticity.
- Government procurement policy in the relevant jurisdiction permits cloud consumption with appropriate controls. There is no policy barrier to cloud adoption that would make on-premises the preferred path.

### Why IBM Cloud Pak for Data (DataStage) over re-platforming

Re-platforming DataStage to Azure Data Factory, AWS Glue, or Apache Spark/Airflow was evaluated. It was deferred — not permanently rejected — on the following grounds:

- The Agency's existing DataStage pipeline estate represents significant embedded institutional knowledge in job design, sequencing, and error-handling logic. The **Knowledge Preservation** concern (C14) is material: a re-platform requires full pipeline revalidation, which carries high risk of silent regression in data correctness.
- IBM Cloud Pak for Data supports migration of existing DataStage jobs with tooling-assisted conversion, reducing revalidation scope.
- A re-platform decision requires a separate options analysis, trade-off matrix, and ADR. It is not appropriate to bind a re-platform decision under hardware-failure time pressure.
- If the Agency determines post-Phase 1 that DataStage on Cloud Pak for Data does not meet cost or capability requirements, that decision will be made with full operational data from the cloud environment — not under duress from failing hardware.

### Why Cognos on Cloud (SaaS or Cloud Pak for Data) over re-platforming

Re-platforming Cognos to Power BI, Tableau, or Qlik was evaluated and deferred on the same knowledge preservation rationale. Additionally:

- The Agency's Cognos report estate contains complex row-level security configurations, multi-dimensional models (OLAP cubes), and parameterised report specifications that do not have direct equivalents in alternative BI platforms. Migration would require report reauthoring, user retraining, and extended parallel run — none of which is compatible with hardware decommission timelines.
- IBM Cognos on Cloud preserves the existing semantic layer and report specifications. This is the only migration path that meets the constraint of no forced reauthoring at go-live.

### Why lift-and-shift for Phase 1

Architectural optimisation during migration introduces two independent risk vectors simultaneously: platform change and design change. Separating them is standard risk management practice. Phase 1 establishes a known-good cloud baseline. Phase 2 optimises from that baseline with operational data. The **Integrity Gap** is addressed in Phase 1 by re-establishing architecture truth on the cloud platform and instrumenting fitness functions from day one of cloud operation.

## Alternatives Considered

| Alternative | Assessment | Disposition |
|-------------|------------|-------------|
| On-premises hardware refresh (like-for-like) | Extends lifecycle of a diverging platform. Does not resolve Integrity Gap. Increasing capital cost trajectory. IBM feature investment diverging from on-prem. | Rejected |
| On-premises refresh with OS/middleware modernisation | Partial improvement. Does not address hardware refresh cost cycle, IBM product direction, or absence of fitness function coverage. | Rejected |
| Re-platform DataStage to Azure Data Factory | High revalidation risk. Requires separate options analysis. Not compatible with hardware decommission timeline. | Deferred — ADR-00X to be raised if initiated |
| Re-platform Cognos to Microsoft Power BI | Significant reauthoring required. Row-level security migration complexity high. Not compatible with go-live constraint. | Deferred — ADR-00X to be raised if initiated |
| IBM Cognos Analytics on Cloud (SaaS) | Viable. Subject to data sovereignty confirmation. Removes infrastructure management obligation. Preferred if sovereignty requirements are met. | Under evaluation — Trade-off Matrix required (CR-T2) |
| IBM Cloud Pak for Data (self-managed on cloud IaaS) | Viable. Provides maximum control over data residency and configuration. Higher operational burden than SaaS. | Under evaluation — Trade-off Matrix required (CR-T2) |
| Migrate to a third-party cloud data platform (Snowflake + Sigma/Tableau) | Full re-platform. All risk vectors open simultaneously. Not appropriate under current constraints. | Rejected for this phase |

## Consequences

### Positive Consequences

- **Hardware failure risk eliminated** for the DataStage and Cognos workloads upon Phase 1 go-live. The primary operational risk driving this decision is resolved.
- **Integrity Gap addressed at migration boundary.** The migration event is the forcing function to document current-state pipelines and report specifications accurately, re-establishing architecture truth before cloud deployment.
- **Fitness function coverage established from day one of cloud operation.** The cloud environment supports continuous assertion in a way the legacy on-premises environment cannot.
- **IBM product roadmap alignment restored.** The Agency will have access to AI-assisted ETL features, automated data quality, and enhanced lineage once on Cloud Pak for Data.
- **Elastic capacity** removes the current constraint blocking new reporting and pipeline initiatives.

### Negative Consequences and Risks

| Risk | Description | Mitigation |
|------|-------------|------------|
| **Data sovereignty non-compliance** | Cloud deployment of government data requires confirmed jurisdiction-resident data storage. If the selected cloud provider cannot confirm in-jurisdiction data residency, this ADR is blocked. | Sovereignty confirmation is a precondition for cloud provider selection. Must be resolved before ADR moves to Accepted status. |
| **Hidden pipeline complexity** | The Integrity Gap in the current environment means undocumented DataStage jobs, undocumented dependencies, and undocumented transformation logic may exist. Discovery during migration will extend Phase 1 timeline. | Pre-migration discovery sprint is mandatory. Pipeline inventory and dependency mapping must be completed before migration execution begins. |
| **Report fidelity regression** | Cognos report specifications, particularly those using legacy Report Studio features or Framework Manager namespace configurations, may behave differently on the cloud version. | Parallel run period with structured UAT against production report outputs is mandatory before on-premises decommission. |
| **Operational model change** | Cloud Pak for Data (OpenShift-based) requires different operational skills than traditional on-premises IBM middleware. The Agency's operations team will require upskilling. | Training and knowledge transfer plan required as part of Phase 1 delivery scope. |
| **Cost model shift** | Transition from capital expenditure (hardware) to operational expenditure (cloud consumption) requires finance and procurement model adjustment. Cloud cost governance (tagging, budgets, anomaly alerts) must be established at deployment. | Cloud cost fitness functions (see below) are mandatory from Phase 1 go-live. |
| **Phase 2 deferral becomes permanent** | Lift-and-shift Phase 1 without committed Phase 2 delivery risks optimisation debt accumulating on the new platform — replicating the current problem in a cloud environment. | Phase 2 initiation must be scheduled and committed before Phase 1 go-live. ADR-00X for pipeline and semantic layer modernisation to be raised within 60 days of Phase 1 go-live. |

## Fitness Functions

All fitness functions below must be active in the cloud environment before on-premises decommission is approved. Each carries a reference per **CR-T4**.

---

### FF-001 — DataStage Pipeline Completion Rate

```python
# [ADR-001] — DataStage pipeline SLA: all scheduled jobs must complete successfully
# Measures rate of successful pipeline execution against scheduled runs
assert pipeline_completion_rate(window="24h") >= 0.995
# Threshold: 99.5% of scheduled DataStage jobs complete without failure
# Failure action: alert on-call data engineer; block downstream Cognos refresh
```

### FF-002 — DataStage Pipeline Latency (Critical Path)

### FF-003 — Cognos Report Render Time (p95)


### FF-004 — Data Sovereignty Boundary Enforcement
